# JTA Terraform Infrastructure 

## Overview
This Terraform project provisions a private Azure web application platform with network segmentation and secure data access.

What is deployed:
- Resource Group
- Virtual Network (`10.0.0.0/16`)
- Subnets:
  - `GatewaySubnet` (`10.0.0.0/25`)
  - App subnet (`10.0.10.0/24`)
  - DB subnet (`10.0.20.0/24`)
- NSGs for app/db subnet boundaries
- VPN Gateway (for corporate/VPN entry)
- App Service Plan + 2 Linux Web Apps (zone-tagged)
- PostgreSQL Flexible Server + database (private access)
- Blob Storage account + private container (private endpoint)
- Key Vault (private endpoint)
- Private DNS zones + VNet links for:
  - Blob: `privatelink.blob.core.windows.net`
  - PostgreSQL: `privatelink.postgres.database.azure.com`
  - Key Vault: `privatelink.vaultcore.azure.net`


## Project structure
- `main.tf`: root composition and module wiring
- `variables.tf`: configurable inputs
- `outputs.tf`: exposed outputs
- `modules/`: domain modules (`networking`, `compute`, `database`, `storage`, `keyvault`, `vpn`)

## Prerequisites
- Terraform `>= 1.0` (tested with 1.14.x)
- Azure subscription access with permissions to create network, compute, data, and key management resources
- Azure authentication configured (Azure CLI login or service principal env vars)

Example Azure login:
```bash
az login
az account set --subscription "<SUBSCRIPTION_ID_OR_NAME>"
```


## How to run / plan
From the `JTA` directory:

```bash
terraform init
terraform fmt -recursive
terraform validate
terraform plan -out tfplan
```

Apply:
```bash
terraform apply tfplan
```

Destroy (if needed):
```bash
terraform destroy
```

## Useful outputs
After apply, key outputs include:
- VNet/subnet IDs
- VPN gateway ID/public IP
- App Service names/hostnames
- PostgreSQL server FQDN
- Storage account/container name
- Key Vault URI
- Private endpoint IPs

## Notes
- DB admin password is generated by Terraform (`random_password`).
- In production, use a remote encrypted backend and strict state access controls.
